# SAGP App Contract Freeze

This document freezes the handoff contracts among the three SAGP applications.
It is intentionally narrow: new features should conform to these contracts until
there is an explicit migration.

## Application boundaries

```text
App 1: raw CSV exports -> output/SAGP_Reconciliation.xlsx
App 2: output/SAGP_Reconciliation.xlsx:Master -> output/sagp_members.db
App 3: GUI reads/edits output/sagp_members.db
```

After App 2 builds the SQLite database, the SQLite database is the operational
source of truth. The workbook remains a reproducible migration/audit artifact,
not the object edited by App 3.

## App 1 -> App 2 workbook contract

App 2 reads the `Master` sheet from `SAGP_Reconciliation.xlsx`.

Required columns, in logical contract order:

```text
PersonID
DisplayName
FirstName
MiddleName
LastName
Suffix
Institution
Title
PrimaryEmail
Phone
City
StateProvince
PostalCode
Country
OriginalMembershipCode
CodeHistory
AppearsIn
```

Column meaning:

- `PersonID`: stable SAGP person key generated by App 1; primary key in SQLite.
- `DisplayName`: non-empty human display label. If missing, App 2 falls back to `PersonID` and records a warning.
- `FirstName`, `MiddleName`, `LastName`, `Suffix`: interpreted name fields.
- `Institution`, `Title`, `PrimaryEmail`, `Phone`, `City`, `StateProvince`, `PostalCode`, `Country`: interpreted contact fields.
- `OriginalMembershipCode`: current/best preserved membership code from the reconciliation pass; not semantically interpreted by App 2.
- `CodeHistory`: semicolon/comma-delimited code history. App 2 splits this into `membership_code_history`.
- `AppearsIn`: semicolon/comma-delimited source file history. App 2 splits this into `source_appearances`.

App 2 may ignore additional workbook columns. App 1 may add columns only if the
required columns above remain present and retain their meanings.

## App 2 -> App 3 SQLite contract

Canonical table: `members`.

```sql
person_id TEXT PRIMARY KEY,
display_name TEXT NOT NULL,
title TEXT,
first_name TEXT,
middle_name TEXT,
last_name TEXT,
suffix TEXT,
institution TEXT,
primary_email TEXT,
secondary_email TEXT,
phone TEXT,
address1 TEXT,
address2 TEXT,
city TEXT,
state_province TEXT,
postal_code TEXT,
country TEXT,
region TEXT,
membership_status TEXT,
original_membership_code TEXT,
member_since TEXT,
last_paid_year INTEGER,
active INTEGER NOT NULL DEFAULT 1,
notes TEXT,
created_at TEXT NOT NULL,
updated_at TEXT NOT NULL
```

Supporting tables:

- `source_appearances(person_id, source_file)` records workbook/source provenance.
- `membership_code_history(person_id, code)` records preserved historical codes.
- `schema_info(key, value)` records database metadata.
- `import_log(...)` records each App 2 import.

App 3 must use `state_province`, not `state`; `original_membership_code`, not
`membership_code`; and must not assume `department` exists.

## Current edit policy

App 3 may display the database immediately. Editing remains conservative:
contact/status fields should be edited only through fields that exist in the
`members` table above. Provenance/history tables should be treated as read-only
unless a later migration explicitly defines write behavior.

## Migration rule

Any incompatible change requires:

1. updating this document,
2. incrementing the contract/version metadata,
3. adding a migration or rebuild instruction, and
4. updating App 3 schema validation before enabling the changed behavior.

## App 3 internal database API contract

App 3 now treats `database_api.SagpDatabase` as its internal database boundary.
The GUI, future dashboards, and future AI-assisted tools should call this named
API rather than issuing SQLite directly.  App 2 remains responsible for creating
the SQLite schema described above; App 3 is responsible for validating and
accessing that schema through its API layer.

Current named API surface:

```python
SagpDatabase(path)
.close()
.validate_schema()
.get_summary()
.get_schema_info()
.count_members()
.list_members(limit=None, offset=0)
.get_member(person_id)
.search_members(text, limit=None)
.get_source_appearances(person_id)
.get_membership_code_history(person_id)
.update_member(member)
```
